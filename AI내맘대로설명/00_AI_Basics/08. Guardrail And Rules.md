# 08. Guardrail And Rules

# 왜 AI에는 Rule과 Guardrail이 필요할까?

이전 글에서는:

```text
AI Agent
```

에 대해 알아봤다.

AI Agent는 단순히 답변만 하는 것이 아니라:

- 검색하고
    
- Tool을 사용하고
    
- 실행하고
    
- 다시 판단하는
    

행동 중심 구조라고 설명했다.

그런데 여기서 아주 중요한 문제가 등장한다.

```text
AI를 정말 마음대로 행동하게 둬도 괜찮을까?
```

실제로 현재 AI 시스템에서 가장 중요하게 보는 영역 중 하나가 바로:

- Rule
    
- Guardrail
    
- 제한 정책
    
- 통제 시스템
    

같은 구조들이다.

---

# AI는 생각보다 "통제"가 중요하다

많은 사람들이 AI를 보면:

```text
엄청 똑똑하니까
알아서 잘 행동하겠지?
```

라고 생각한다.

하지만 실제 AI는:

```text
확률 기반으로 행동
```

하는 경우가 많다.

즉 조건이 애매하면:

- 이상한 방향으로 가거나
    
- 엉뚱한 자료를 찾거나
    
- 반복 행동을 하거나
    
- 비용을 계속 사용
    

하는 일이 실제로 발생한다.

---

# 예시로 이해해보기

예를 들어 AI에게:

```text
코끼리를 찾아줘
```

라고 요청했다고 해보자.

사람은 보통:

```text
진짜 동물 코끼리
```

를 떠올릴 가능성이 높다.

하지만 AI 입장에서는:

- 코끼리 장난감
    
- 코끼리 캐릭터
    
- 코끼리 뉴스
    
- 코끼리 이미지
    
- 코끼리 영화
    

같은 것도 모두 관련 있다고 판단할 수 있다.

즉:

```text
조건이 모호하면
AI도 모호하게 행동
```

할 수 있다.

---

# AI는 반복 행동도 할 수 있다

이건 실제로 자주 발생하는 문제다.

예를 들어 AI가:

```text
검색
↓
결과 부족 판단
↓
다시 검색
↓
결과 부족 판단
↓
다시 검색
```

같은 루프를 계속 돌 수 있다.

사람 입장에서는:

```text
왜 같은 걸 계속 하지?
```

싶지만,

AI는:

```text
현재 목표를 달성하려고 계속 시도
```

하는 구조에 가까울 수 있다.

---

# 비용 문제도 매우 중요하다

현재 대부분의 AI는:

```text
사용량 기반 비용
```

이 발생한다.

예를 들어:

- AI 호출
    
- 검색 API
    
- 외부 Tool 사용
    
- 긴 문서 처리
    

같은 것들이 모두 비용이 될 수 있다.

그래서 AI가:

```text
의미 없는 반복 작업
```

을 하면 실제 서비스 비용이 빠르게 증가할 수 있다.

---

# 그래서 Rule이 등장한다

Rule은 쉽게 말하면:

```text
AI 행동 규칙
```

이다.

예를 들어:

- 검색은 최대 5번까지만
    
- 동일 검색 반복 금지
    
- 특정 사이트 접근 금지
    
- 위험 명령 차단
    
- 일정 시간 이상 실행 금지
    

같은 것들이다.

즉:

```text
AI가 행동할 수 있는 범위
```

를 정해주는 것이다.

---

# Guardrail은 무엇인가?

Guardrail은 쉽게 말하면:

```text
AI 안전장치
```

에 가깝다.

예를 들어:

- 이상 행동 감지
    
- 위험한 답변 차단
    
- 개인정보 차단
    
- 비용 초과 방지
    
- 승인 요청
    

같은 역할을 한다.

즉 AI가:

```text
너무 위험한 방향으로 가지 않게 막는 구조
```

라고 볼 수 있다.

---

# 실제 AI 시스템은 통제가 굉장히 많다

현재 기업용 AI 시스템은 생각보다:

```text
통제 중심
```

으로 설계되는 경우가 많다.

예를 들어:

- 최대 실행 횟수 제한
    
- Timeout
    
- Retry 제한
    
- 사용자 권한 제한
    
- Tool 사용 제한
    
- 승인 프로세스
    
- 로그 저장
    

같은 구조를 매우 중요하게 본다.

왜냐하면 실제 서비스에서는:

```text
AI 성능
```

만큼이나:

```text
AI 안정성
```

이 중요하기 때문이다.

---

# 사람 승인(Human In The Loop)도 많이 사용한다

특히 중요한 작업에서는:

```text
AI가 바로 실행하지 못하게
```

하는 경우도 많다.

예를 들어:

- 메일 발송
    
- 결제
    
- 운영 서버 수정
    
- 고객 공지
    

같은 작업은 보통:

```text
사람 승인 후 실행
```

구조를 사용한다.

이런 구조를 보통:

```text
Human In The Loop
```

라고 부른다.

---

# 결국 AI도 관리 대상이다

예전에는:

```text
사람이 시스템을 관리
```

했다면,

최근에는:

```text
AI를 관리하는 시스템
```

도 같이 만들기 시작했다.

예를 들어:

- Audit Log
    
- Monitoring
    
- Policy
    
- Runtime 제한
    
- 비용 관리
    

같은 시스템들이다.

즉 AI도 이제는:

```text
운영 관리 대상
```

이 되어가고 있다.

---

# 정리

AI Agent는 매우 강력하지만:

```text
무조건 자유롭게 두면 위험할 수 있다
```

그래서 실제 AI 시스템에서는:

- Rule
    
- Guardrail
    
- Timeout
    
- Approval
    
- Monitoring
    

같은 구조를 함께 사용한다.

즉 현재 AI 시스템은 단순히:

```text
똑똑한 AI 만들기
```

만 중요한 것이 아니라,

```text
안전하게 통제 가능한 AI 만들기
```

도 매우 중요한 시대가 되었다.

다음 글에서는:

```text
실제 회사들은 AI 시스템을 어떻게 구성하고 있을까?
```

를 알아보려고 한다.
