# 06. RAG

# RAG는 무엇인가?

이전 글에서는:

```text
벡터(Vector)
```

개념에 대해 알아봤다.

그리고 벡터를 이용하면:

```text
문장의 의미 기반 검색
```

이 가능하다는 것도 설명했다.

그렇다면 여기서 자연스럽게 이런 생각이 든다.

```text
검색한 자료를
AI에게 같이 보여주면 어떻게 될까?
```

바로 여기서 등장하는 구조가:

```text
RAG
```

이다.

---

# RAG란?

RAG는:

```text
Retrieval Augmented Generation
```

의 약자이다.

이걸 아주 쉽게 말하면:

```text
검색 + AI 답변
```

구조라고 볼 수 있다.

---

# 왜 RAG가 필요할까?

이전 글에서 설명했듯이 LLM은:

- 최신 정보를 모를 수 있고
    
- 회사 내부 문서를 모를 수 있고
    
- 긴 내용을 모두 기억하지 못하고
    
- 환각(Hallucination)이 발생하기도 한다
    

즉:

```text
AI 혼자만으로는
정확한 업무 처리가 어려운 경우
```

가 많다.

그래서 나온 아이디어가:

```text
필요한 자료를 먼저 검색해서
AI에게 참고자료로 같이 전달하자
```

이다.

---

# 가장 기본적인 RAG 흐름

RAG는 보통 다음 순서로 동작한다.

```text
질문
↓
관련 자료 검색
↓
관련 내용 추출
↓
AI에게 전달
↓
답변 생성
```

예를 들어:

```text
질문:
우리 회사 휴가 정책 알려줘
```

↓

```text
휴가 정책 문서 검색
```

↓

```text
관련 문단 추출
```

↓

```text
AI에게 참고자료 전달
```

↓

```text
문서를 기반으로 답변 생성
```

이런 흐름이다.

---

# 여기서 중요한 점

중요한 건:

```text
AI가 원래 알고 있는 게 아니다
```

라는 점이다.

즉:

```text
검색한 자료를 참고해서
답변하는 구조
```

에 가깝다.

그래서 실제 업무에서는:

```text
LLM 단독
```

보다:

```text
RAG 구조
```

를 훨씬 많이 사용한다.

---

# 왜냐하면 회사 데이터는 계속 바뀌기 때문이다

예를 들어 회사에는:

- 정책 문서
    
- 회의록
    
- 공지사항
    
- 매뉴얼
    
- 운영 가이드
    

같은 데이터가 계속 수정된다.

그런데 이런 내용을 매번 AI 모델에 다시 학습시키는 것은 현실적으로 어렵다.

그래서 대부분은:

```text
문서는 따로 저장
↓
필요할 때 검색
↓
AI에게 전달
```

구조를 사용한다.

---

# RAG의 핵심은 "검색 품질"

많은 사람들이:

```text
AI 모델 성능
```

만 중요하다고 생각한다.

하지만 실제 RAG에서는:

```text
무슨 자료를 찾았는가
```

가 훨씬 중요할 때도 많다.

왜냐하면 AI는:

```text
들어온 자료 기반으로 답변
```

하기 때문이다.

즉 검색 결과가 이상하면:

- 엉뚱한 답변
    
- 틀린 답변
    
- 관련 없는 답변
    

이 나올 가능성이 커진다.

---

# 그래서 Chunk라는 개념이 등장한다

문서를 통째로 넣는 것은 비효율적이다.

예를 들어:

```text
PDF 500페이지 전체
```

를 AI에게 그대로 보내는 것은:

- 비용도 크고
    
- 속도도 느리고
    
- 중요한 내용을 놓칠 가능성도 있다
    

그래서 보통 문서를:

```text
작은 단위로 잘라 저장
```

한다.

이 조각을:

```text
Chunk
```

라고 부른다.

---

# Chunk 예시

예를 들어 문서를:

- 문단 단위
    
- 페이지 단위
    
- 제목 기준
    

등으로 잘라 저장할 수 있다.

그리고 질문이 들어오면:

```text
질문과 가장 비슷한 Chunk
```

를 찾아서 AI에게 전달한다.

---

# 그래서 벡터 검색이 중요하다

여기서 이전 글의:

```text
벡터(Vector)
```

개념이 다시 등장한다.

질문도 벡터로 변환하고,

문서 Chunk도 벡터로 변환해서 저장한다.

그리고:

```text
가장 의미가 비슷한 Chunk
```

를 찾는다.

즉 현재 많은 AI 검색 시스템은:

```text
벡터 검색 기반
```

으로 동작한다.

---

# RAG는 비용 절약에도 중요하다

RAG는 단순히 정확도만 높이는 기술이 아니다.

실제로는:

```text
비용 절약
```

에도 매우 중요하다.

왜냐하면:

- 모든 데이터를 AI에게 보내면 비싸고
    
- 같은 질문을 반복하면 비효율적이고
    
- 이미 가진 데이터를 재활용하는 게 더 효율적
    

이기 때문이다.

그래서 실제 시스템에서는:

```text
우리 데이터 먼저 검색
↓
필요한 것만 AI에게 전달
```

구조를 많이 사용한다.

---

# 현재 대부분의 기업 AI는 RAG 기반이다

현재 기업에서 사용하는 AI 시스템 대부분은:

```text
RAG 구조
```

를 어느 정도 포함하고 있다.

왜냐하면 실제 업무에서는:

```text
회사 데이터 기반 답변
```

이 매우 중요하기 때문이다.

즉 단순 챗봇보다:

```text
검색 가능한 AI 시스템
```

으로 발전하고 있는 것이다.

---

# 정리

RAG는 쉽게 말하면:

```text
검색 + AI 답변
```

구조이다.

흐름은 보통:

```text
질문
↓
관련 자료 검색
↓
관련 내용 추출
↓
AI에게 전달
↓
답변 생성
```

으로 동작한다.

그리고 현재 AI 시스템에서는:

```text
검색 품질
```

이 AI 성능만큼 중요해지고 있다.

다음 글에서는:

```text
AI Agent는 무엇인가?
```

를 알아보려고 한다.
