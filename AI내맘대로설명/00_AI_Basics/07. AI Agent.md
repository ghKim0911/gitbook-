# 07. AI Agent

# AI Agent는 무엇인가?

이전 글에서는:

```text
RAG
```

구조에 대해 알아봤다.

RAG는 쉽게 말하면:

```text
검색 + AI
```

구조였다.

그런데 최근에는 여기서 한 단계 더 발전한 형태가 등장하기 시작했다.

바로:

```text
AI Agent
```

이다.

---

# 기존 AI는 보통 "질문 → 답변" 구조였다

예전의 AI 사용 방식은 대부분:

```text
질문
↓
답변
```

구조에 가까웠다.

예를 들어:

```text
오늘 날씨 알려줘
```

↓

```text
오늘 서울은 맑습니다
```

처럼 동작했다.

즉:

```text
한 번 질문하면
한 번 답변
```

하는 구조였다.

---

# 그런데 사람들은 점점 더 복잡한 일을 시키기 시작했다

예를 들어:

```text
회의 내용을 정리해서
노션에 저장하고
슬랙으로 공유해줘
```

같은 요청이다.

이런 작업은 단순 답변만으로는 어렵다.

왜냐하면:

- 문서도 읽어야 하고
    
- 요약도 해야 하고
    
- 외부 시스템도 호출해야 하고
    
- 여러 단계를 순서대로 처리해야 하기 때문이다
    

---

# 그래서 등장한 개념이 AI Agent이다

AI Agent를 아주 쉽게 말하면:

```text
생각만 하는 AI가 아니라
행동까지 하는 AI
```

에 가깝다.

즉 AI가:

- 검색하고
    
- 계획 세우고
    
- Tool 사용하고
    
- 결과를 다시 확인하고
    
- 다음 행동을 결정
    

하기 시작한 것이다.

---

# AI Agent의 기본 흐름

AI Agent는 보통 다음처럼 동작한다.

```text
목표 입력
↓
무엇을 해야 할지 계획
↓
필요한 Tool 선택
↓
실행
↓
결과 확인
↓
다음 행동 결정
```

즉 단순히:

```text
답변 생성
```

만 하는 것이 아니라,

```text
작업 수행
```

까지 하기 시작한 것이다.

---

# Tool이라는 개념

여기서 중요한 개념이 하나 나온다.

바로:

```text
Tool
```

이다.

AI는 기본적으로 외부 시스템에 직접 접근하지 못한다.

그래서 필요한 기능들을:

- 검색 API
    
- 메일 전송
    
- DB 조회
    
- 파일 읽기
    
- 슬랙 전송
    

같은 형태로 연결한다.

그리고 AI는:

```text
지금 어떤 Tool이 필요한가?
```

를 판단해서 사용한다.

---

# 예시로 이해해보기

예를 들어:

```text
오늘 일정 요약해서 메일 보내줘
```

라고 요청했다고 해보자.

AI Agent는:

```text
1. 캘린더 조회
2. 일정 읽기
3. 내용 요약
4. 메일 작성
5. 메일 전송
```

같은 순서로 행동할 수 있다.

즉 단순 챗봇보다 훨씬:

```text
실행 중심
```

구조에 가까워진다.

---

# AI Agent는 여러 번 생각할 수도 있다

기존 챗봇은 보통:

```text
한 번 질문 → 한 번 답변
```

구조였다.

하지만 AI Agent는:

```text
실행 결과를 보고
다시 다음 행동을 결정
```

할 수 있다.

예를 들어:

```text
검색 결과가 부족하네?
```

↓

```text
검색 조건을 바꿔서 다시 검색
```

같은 행동이 가능하다.

---

# 그런데 여기서 새로운 문제가 생긴다

AI가 직접 행동하기 시작하면:

- 이상한 방향으로 가거나
    
- 같은 작업을 반복하거나
    
- 무한 루프를 돌거나
    
- 비용이 계속 증가하거나
    
- 위험한 행동을 수행
    

할 가능성도 생긴다.

예를 들어:

```text
코끼리를 찾아줘
```

라고 했는데 AI가:

- 관련 없는 사이트를 계속 검색하거나
    
- 의미 없는 반복 검색을 하거나
    
- 엉뚱한 데이터를 모으는 경우
    

도 발생할 수 있다.

---

# 그래서 AI를 통제하기 시작했다

이 문제를 해결하기 위해 등장한 것이:

- Rule
    
- Guardrail
    
- Timeout
    
- Approval
    
- Retry 제한
    

같은 개념들이다.

즉 AI를:

```text
무조건 자유롭게 두는 것
```

이 아니라,

```text
정해진 범위 안에서 행동하게 만드는 구조
```

가 중요해졌다.

---

# 최근 AI 시스템은 대부분 Agent 방향으로 발전 중이다

현재 AI 업계는 단순 챗봇보다:

```text
작업 수행형 AI
```

방향으로 빠르게 발전하고 있다.

예를 들어:

- 업무 자동화
    
- 문서 처리
    
- 운영 자동화
    
- 고객 응대
    
- 데이터 분석
    

같은 영역에서 AI Agent 활용이 늘어나고 있다.

---

# 하지만 아직 완벽하지는 않다

AI Agent는 매우 강력하지만 아직 한계도 많다.

예를 들어:

- 잘못된 판단 가능
    
- 반복 행동 가능
    
- 예상 못한 행동 가능
    
- 비용 폭주 가능
    
- 보안 문제 가능
    

같은 문제가 있다.

그래서 실제 서비스에서는 보통:

```text
AI + Rule + 사람 검토
```

구조를 함께 사용한다.

---

# 정리

AI Agent는 쉽게 말하면:

```text
답변만 하는 AI가 아니라
직접 행동하는 AI
```

이다.

즉 AI가:

- 계획하고
    
- Tool을 사용하고
    
- 실행하고
    
- 결과를 확인하고
    
- 다시 행동을 결정
    

하기 시작한 구조이다.

그리고 여기서 매우 중요한 문제가 등장한다.

```text
AI를 어디까지 믿고
어떻게 통제할 것인가?
```

다음 글에서는:

```text
왜 AI에는 Rule과 Guardrail이 필요한가?
```

를 알아보려고 한다.
