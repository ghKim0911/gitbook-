# 09. Real World AI System

# 실제 회사들은 AI 시스템을 어떻게 만들고 있을까?

지금까지 다음 개념들을 차례대로 알아봤다.

- AI
    
- LLM
    
- LLM의 한계
    
- 검색 시스템
    
- 벡터(Vector)
    
- RAG
    
- AI Agent
    
- Guardrail
    

그렇다면 이제 자연스럽게 이런 궁금증이 생긴다.

```text
실제 회사들은
AI 시스템을 어떻게 만들고 있을까?
```

이번 글에서는 지금까지 설명했던 개념들이 실제로 어떻게 연결되는지 알아보려고 한다.

---

# 예전 AI 시스템은 단순 챗봇 형태가 많았다

초기에는 대부분:

```text
질문
↓
AI 답변
```

형태에 가까웠다.

예를 들어:

- 고객 문의 응답
    
- FAQ 챗봇
    
- 간단한 검색 봇
    

같은 구조였다.

하지만 실제 업무에 적용하려고 하자 여러 문제가 생기기 시작했다.

---

# 실제 업무에서는 회사 데이터를 알아야 한다

예를 들어 회사에서 AI를 사용하려면:

- 사내 정책
    
- 운영 문서
    
- 회의록
    
- 고객 정보
    
- 매뉴얼
    
- 업무 프로세스
    

같은 데이터를 알아야 한다.

그런데 일반 LLM은:

```text
회사 내부 데이터를 기본적으로 모른다
```

그래서 등장한 구조가:

```text
RAG
```

였다.

---

# 그래서 검색 시스템이 함께 붙기 시작했다

현재 많은 회사 시스템은 보통 다음처럼 구성된다.

```text
문서 저장
↓
문서 분리(Chunk)
↓
벡터 변환(Embedding)
↓
벡터 DB 저장
```

그리고 질문이 들어오면:

```text
질문
↓
벡터 검색
↓
관련 문서 찾기
↓
AI에게 전달
↓
답변 생성
```

구조로 동작한다.

즉:

```text
검색 + AI
```

형태가 현재 기업 AI의 기본 구조가 되기 시작했다.

---

# 그런데 여기서 또 발전하기 시작했다

사람들은 단순 답변보다:

```text
실제 작업 수행
```

을 원하기 시작했다.

예를 들어:

- 메일 보내기
    
- 회의 정리
    
- 일정 등록
    
- 데이터 분석
    
- 운영 자동화
    

같은 것들이다.

그래서 등장한 것이:

```text
AI Agent
```

구조이다.

---

# 현재 AI 시스템은 여러 구성 요소가 함께 동작한다

최근 AI 시스템은 보통:

- 검색 시스템
    
- 벡터 DB
    
- LLM
    
- AI Agent
    
- Tool 시스템
    
- Guardrail
    
- Monitoring
    

같은 것들이 함께 동작한다.

즉 이제 AI는 단순 모델 하나가 아니라:

```text
여러 시스템이 연결된 플랫폼 형태
```

로 발전하고 있다.

---

# 실제 흐름 예시

예를 들어 사용자가:

```text
이번 주 회의 내용 정리해서 메일 보내줘
```

라고 요청했다고 해보자.

현재 AI 시스템은 보통:

```text
1. 회의록 검색
2. 관련 내용 추출
3. AI 요약
4. 메일 생성
5. 승인 요청
6. 메일 발송
7. 로그 저장
```

같은 흐름으로 동작할 수 있다.

즉 단순 답변보다:

```text
업무 자동화 플랫폼
```

에 가까워지고 있다.

---

# 그래서 운영 시스템이 중요해졌다

AI가 실제 업무를 수행하기 시작하면서:

- 비용 관리
    
- 권한 관리
    
- 실행 로그
    
- 승인 프로세스
    
- 위험 행동 차단
    

같은 것들이 매우 중요해졌다.

예를 들어 회사에서는 보통:

- 누가 실행했는가
    
- 어떤 Tool을 사용했는가
    
- 얼마의 비용이 발생했는가
    
- 어떤 데이터에 접근했는가
    

같은 기록을 남긴다.

즉 AI도 이제:

```text
운영 관리 대상 시스템
```

이 된 것이다.

---

# 그래서 Runtime 개념도 등장한다

최근에는 단순히 AI 모델만 보는 것이 아니라:

```text
AI가 어떻게 실행되는가
```

를 관리하는 Runtime 개념도 중요해지고 있다.

예를 들어:

- 최대 실행 횟수
    
- Timeout
    
- Retry 정책
    
- Tool 사용 제한
    
- 비용 제한
    
- 병렬 실행 제한
    

같은 것들을 관리한다.

즉 AI를:

```text
안전하고 통제 가능하게 운영
```

하는 방향으로 발전하고 있다.

---

# 최근 AI 업계 흐름

현재 AI 업계는 크게 다음 방향으로 발전 중이다.

```text
단순 챗봇
↓
RAG
↓
AI Agent
↓
업무 자동화
↓
AI 운영 플랫폼
```

즉 이제는 단순히:

```text
AI가 얼마나 똑똑한가
```

보다:

```text
AI를 실제 업무에 어떻게 연결할 것인가
```

가 더 중요해지고 있다.

---

# 결국 중요한 것은 "연결"이다

현재 실제 AI 시스템은 보통:

- LLM
    
- 검색 시스템
    
- 회사 데이터
    
- Tool
    
- 권한 시스템
    
- 운영 정책
    

같은 것들을 연결해서 만든다.

즉 AI 하나만으로 동작한다기보다:

```text
여러 시스템을 연결한 구조
```

에 가깝다.

---

# 정리

현재 기업 AI 시스템은 보통:

```text
검색 + LLM + Agent + 운영 시스템
```

구조로 발전하고 있다.

즉 이제 AI는 단순 챗봇을 넘어서:

```text
실제 업무를 수행하는 플랫폼
```

방향으로 발전하고 있는 것이다.

그리고 앞으로는:

- 더 강한 Agent
    
- 더 정교한 검색
    
- 더 안전한 Guardrail
    
- 더 큰 자동화 시스템
    

방향으로 계속 발전할 가능성이 높다.
